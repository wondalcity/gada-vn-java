## 7. Critical Isolation Rules

These rules prevent accidental data cross-contamination between environments.

| Rule | Variable | Why |
|------|----------|-----|
| **R1** | `AWS_BUCKET` / `AWS_BUCKET` | Different S3 buckets per env. Staging bucket: `gada-vn-staging-uploads`. Using prod bucket from staging would expose real user PII to test writes. |
| **R2** | `FIREBASE_PROJECT_ID` | Different Firebase projects per env. Staging workers/managers must not appear in production Firebase Auth user list. |
| **R3** | `DB_HOST` | Different RDS instances. Staging test data (fake workers, test contracts) must not pollute production. |
| **R4** | `ENCRYPTION_KEY` | Different AES keys per env. Signatures encrypted in staging cannot be decrypted in production and vice versa. |
| **R5** | `EXPO_PUBLIC_API_URL` | Mobile app must always call staging API from staging build and production API from production build. Leaking production API URL into a preview build exposes real user data. |
| **R6** | `NEXT_PUBLIC_API_BASE_URL` | Same reasoning as R5 â€” embedded at build time. |
| **R7** | `APP_DEBUG` | Must be `false` in staging and production â€” never expose stack traces to users. |
| **R8** | `ADMIN_PANEL_PASSWORD` | Must not use default value (`gadaAdmin2026!`) in any deployed environment. |
